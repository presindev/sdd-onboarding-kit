Purpose

This document defines acceptance-level test scenarios for the feature.

These are not necessarily final automated test implementations. They describe the observable behavior that must be true for the feature to be accepted.

The acceptance tests must verify:

  • main success scenarios;
  • validation rules;
  • edge cases;
  • error cases;
  • non-goals that must remain unchanged.

Acceptance tests

AT1 Default limit shows five notes, newest first Implemented
Related requirements
REQ-001
Scenario
A user with several notes asks for recent notes without options.
Preconditions
Store contains 8 notes with distinct created_at values.
Action
Run notes recent.
Expected result
Exactly 5 notes printed, ordered newest first; exit code 0.
Negative expectations
No note older than the 5 newest appears.

Manual acceptance checklist

Use during human review.

  • All requirements have at least one acceptance test.
  • Main success path is covered.
  • Empty state is covered, if relevant.
  • Invalid input is covered, if relevant.
  • Permission / security behavior is covered, if relevant.
  • Non-goals are protected, if relevant.
  • Tests are feasible with the current project test setup.
  • No acceptance test depends on hidden assumptions.
  • Any unresolved behavior is listed in open-questions.html.

Notes for Claude Code

Rules Claude Code must follow

  1. Acceptance tests must map back to requirements.html.
  2. Do not invent behavior that is absent from the functional brief or approved assumptions.
  3. If a test requires an unresolved decision, add or update open-questions.html.
  4. The implementer must create automated tests based on this document where feasible.
  5. The reviewer must verify that implemented tests cover this document.
  6. If an acceptance test cannot be automated, mark it clearly and explain why.
