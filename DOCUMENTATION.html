SDD Onboarding Kit — Documentation

Complete reference for installation, daily workflow, subagents, hooks, and MCP integrations.

1. What is SDD

Spec Driven Development (SDD) is a workflow where Claude Code writes and gets approval for a specification before touching any production code. The spec covers:

  • Requirements — observable behavior the feature must satisfy.
  • Design — which files to change, how, and why.
  • Tasks — an ordered, checkable implementation checklist.

The developer reviews the spec and explicitly approves it. Claude then implements strictly against the approved spec. An automated reviewer validates that the result matches the spec, requirements are traceable, and tests pass.

The goal is to eliminate ambiguity, prevent scope creep, and give the developer a meaningful decision point before code is written.

2. What this kit is

This kit is a reusable installer. It produces a project-specific SDD harness, not a global configuration.

When you run the onboarding prompt, Claude Code reads the kit, inspects your repository, asks you project-specific questions, and generates:

  • A CLAUDE.md tailored to your project.
  • Four Claude subagents (leader, spec-author, implementer, reviewer).
  • The full SDD workflow skill.
  • Local task and spec storage.
  • Optional hooks and MCP integrations.
  • Validation scripts.

The kit itself stays inert. Only the generated harness in your project becomes active.

3. Installation

Prerequisites

  • Claude Code (CLI, desktop app, or IDE extension).
  • A git repository where you want SDD installed.

Step 1 — Add the kit to your project

Copy or clone this kit folder into your project root:

my-project/
└── sdd-onboarding-kit/

Step 2 — Open Claude Code in your project

Open Claude Code with my-project/ as the working directory.

Step 3 — Run the onboarding prompt

Read `sdd-onboarding-kit/instructions.md` and configure this repository to use Spec Driven Development. Ask me all necessary questions before making project-specific decisions.

Step 4 — Answer the configuration questions

Claude Code will inspect your repository and ask about scope, approval policy, requirements format, task storage, state machine, test commands, git policy, hooks, MCPs, and protected files.

Step 5 — Review and confirm

Claude generates a summary of files created, hooks enabled, and any open TODOs. Review the generated CLAUDE.md and .claude/ before starting work.

4. Daily workflow

Start a new feature

Create a new SDD task for: <description>. First generate requirements/design/tasks and do not implement anything until I approve the spec.

Claude will create a task in tasks.json, create specs/<slug>/requirements.html, design.html, and tasks.html, then stop for your review.

Review and approve a spec

Open the HTML spec files under specs/<feature-slug>/ in a browser. When satisfied:

I have reviewed the spec for `<feature-slug>` and I approve it. Change the status to `human_approved` and prepare the implementation following `tasks.html`.

Implement an approved spec

Implement feature `<feature-slug>` following the approved spec strictly. Run tests and leave traceability of what you changed.

Review an implementation

Use the reviewer to validate the implementation of `<feature-slug>` against requirements.html, design.html and tasks.html. Run tests and tell me if it can be marked as done.

5. Task lifecycle

Every SDD task moves through a defined state machine:

pending→ spec_draft→ spec_ready→ human_approved→ in_progress→ review→ done

Optional states: blocked, rejected.

StateMeaningWho can advance it
pending Task exists, no spec yet Claude (starts drafting)
spec_draft Spec is being written or revised Claude
spec_ready Spec complete, awaiting approval Claude (sets after finishing spec)
human_approved Developer approved the spec Developer only
in_progress Implementation underway Claude
review Implementation complete, under validationClaude
done Complete Reviewer (after passing tests)
blocked Waiting on external input Either party
rejected Review found blocking defects Reviewer

6. Spec format

Each feature gets a directory under specs/:

specs/<feature-slug>/
├── requirements.html
├── design.html
├── tasks.html
├── assumptions.html       (when derived from a functional document)
├── open-questions.html    (when derived from a functional document)
├── acceptance-tests.html  (when derived from a functional document)
├── review.html            (written by the reviewer)
├── spec.css               (shared stylesheet)
└── spec.js                (collapsible sections, TOC, tabs)

requirements.html

Defines observable behavior. Supported formats: EARS, user stories, Given/When/Then, numbered requirements.

design.html

Translates requirements into a technical plan. Contains files to change, interfaces, data model changes, test design, risks, rollback plan, and an inline SVG data-flow diagram.

tasks.html

An ordered milestone view with status badges. Each task references requirements. Test and validation tasks are included.

review.html

Written by the reviewer. Contains a color-coded traceability table (blocking / warning / ok), commands run, findings, and a final decision.

7. Subagents

The harness installs four Claude subagents under .claude/agents/:

Leader

The routing advisor. Reads task state and returns which specialist the main conversation should invoke next (subagents cannot invoke other subagents in Claude Code, so orchestration happens in the main conversation, guided by the sdd-workflow skill). Never writes production code, recommends stopping for human approval at spec_ready.

Spec Author

Writes and revises specs. Produces requirements.html, design.html, tasks.html. When working from a functional document also produces assumptions.html, open-questions.html, and acceptance-tests.html.

Implementer

The only agent allowed to write production code. Implements in task order, adds tests, runs validation, marks tasks complete. Does not deviate from the spec without asking.

Reviewer

Validates that the implementation satisfies the spec. Checks requirement coverage, test coverage, design adherence, conventions. Writes review.html with one of: approved, rejected, needs_changes.

8. Hooks

Hooks are optional shell scripts that Claude Code executes automatically. Hooks are never enabled without your explicit approval.

13. Documentation model

The kit's documentation is layered so that each audience reads only what it needs, and so that CLAUDE.md files stay short. Each layer links down to the next instead of duplicating it.

LayerLocationAudience and purpose
Quick start README.md New users: what the kit does, how to install it, the default safety model.
Operational documentation DOCUMENTATION.html (this page) Maintainers and daily users: full installation, workflow, features, and policies.
Theory and policy reference/ Deeper background material: SDD theory, harness engineering, Claude Code primitives, context economy.
Procedures (skills) skills/ → installed as .claude/skills/ Reusable step-by-step procedures, e.g. the SDD workflow skill. Long procedures live here, never in CLAUDE.md.
Roles (agents) agents/ → installed as .claude/agents/ Role-specific subagent definitions: leader, spec-author, implementer, reviewer.

Related entry points: onboarding instructions, onboarding questions, hooks policy, MCP policy.

The same hierarchy applies to the harness generated in a target project: the project's CLAUDE.md stays concise and links to the project's skill, agents, specs, and decision records instead of embedding them.

14. Language policy

English is canonical for all reusable kit files: instructions, questions, reference material, agents, skills, hooks, MCP notes, templates, and this documentation. This keeps the kit portable across teams and repositories.

Generated project-specific artifacts may use the target project's preferred language. When the harness is installed, the developer can choose the language for generated specs, decision records, task descriptions, and history entries. The structure (file names, state names, status values such as spec_ready or human_approved) always stays in English so hooks and scripts keep working.

  • Contributions to the kit itself must be in English (see the contributing guidelines in README.md).
  • If non-English material is ever kept in the kit for historical reasons, it must be clearly marked as non-canonical/legacy.

15. Context economy

Claude Code's context window is a scarce, shared resource: project instructions, file contents, tool definitions, and conversation history all compete for the same space. The kit turns the well-known context-management tricks into one explicit policy, documented in full in reference/context-economy.md. The generated CLAUDE.md carries only a short summary of these rules and links out.

Good vs bad context loading

SituationBad (context-expensive)Good (context-economical)
Project orientationPaste the directory tree and ten files into chatLink a concise project map from CLAUDE.md
Long procedureCopy the full SDD workflow into CLAUDE.mdKeep it in the sdd-workflow skill; it loads only when used
Finding codeRead 20 files in the main conversationDelegate to a subagent; keep only paths and conclusions
Task switchContinue in the same long session/compact between unrelated tasks (or /clear if truly unrelated)
External toolsEnable every MCP globallyConfigure per project; scope heavy toolsets to a dedicated subagent
Past decisionsRely on chat historyRecord in decision logs and specs, link from CLAUDE.md

CLAUDE.md length and linked documentation

  • CLAUDE.md is loaded into every session, so every line has a permanent cost. Keep it to stable facts and short rules.
  • Link to deeper files (reference docs, skills, project map, decision logs) instead of duplicating their content.
  • If a section grows past a screen, move it into a linked file or a skill.

When to compact, spawn subagents, or use skills

ToolUse when
/context A session has been long and behavior degrades; deciding whether to compact; auditing what consumes the window (it shows usage as a grid with optimization suggestions).
/compact [instructions] Between unrelated tasks or after a noisy investigation. Prefer selective compacting: pass focus instructions to preserve decisions, task status, architecture constraints, and unresolved questions.
Subagents Noisy research that reads many files but yields a short conclusion (exploration, surveys, log triage). Not for small targeted reads.
Skills Repeatable multi-step procedures that would otherwise be pasted into chat or bloat CLAUDE.md; a skill's body loads only when invoked.

In an SDD project, compaction and session resets are safe by design: durable truth lives in tasks.json, specs/, decision logs, and history.html, not in chat history. If something matters beyond the session, write it to an artifact before compacting.

Command behavior verified against the Claude Code commands reference (2026-06): /context [all], /compact [instructions].

16. Project map

Onboarding generates a project map: a one-to-two-screen orientation artifact describing where things live in the target repository. It is the context-economy answer to "how does Claude learn the repository structure?" — instead of embedding a directory tree in CLAUDE.md or reading many files at session start, Claude reads one concise, linked artifact.

Location and generation

  • Default location: .claude/context/project-map.md (harness-internal). Teams that want it visible in their docs may choose docs/project-map.md instead (onboarding question, questions.md §12).
  • Generated during onboarding from the Phase 1 repository inspection, using templates/project-map.md.template.
  • If the developer defers generation, onboarding records a clear TODO instead of creating an empty file.
  • The generated CLAUDE.md links to the map; it never embeds the tree.
  • Unknown commands are recorded as TODO: ask the developer — never invented. Secrets, credentials, and tokens are never recorded.

Contents

The map has eight sections: directory tree (2–3 levels, annotated); key entrypoints; package/build/test/lint commands; framework and runtime assumptions; important documentation; protected areas; SDD artifact locations (tasks, specs, history, decisions); and generated or do-not-edit files.

Example

# acme-shop — project map

## Directory tree
src/            application code
  api/          REST endpoints (Express routers)
  domain/       business logic, no I/O
  db/           Prisma schema + repositories
tests/          Vitest suites, mirrors src/
prisma/         migrations (PROTECTED)
public/         static assets

## Key entrypoints
| src/server.ts | HTTP server bootstrap |
| src/api/index.ts | router registration |

## Commands
| Tests | npm test | Lint | npm run lint |
| Typecheck | npm run typecheck | Build | npm run build |

## Framework and runtime assumptions
Node 22, Express 5, Prisma 6, PostgreSQL 16 (docker compose).

## Protected areas
prisma/migrations/, .env*, .github/workflows/

## SDD locations
tasks.json · specs/<feature-slug>/ · history.html · decisions/

## Generated files and do-not-edit
| src/generated/** | regenerate via npm run codegen, never edit |

When to update the map

The map carries its own maintenance rule. Update it when the structure changes significantly:

  • a top-level or load-bearing directory is added, renamed, or removed;
  • an entrypoint moves;
  • build/test/lint commands change;
  • a new protected area or generated path appears.

Do not update it for routine file additions inside existing directories — a map that changes on every commit defeats its purpose, and one that lists every file defeats context economy (see §15).

17. Skill packs

The kit's procedural knowledge is modular. The core sdd-workflow skill is always installed; nine optional skill packs under skills/optional/ are installed only when the developer selects them during onboarding (questions.md §14) or asks later. This keeps the installed harness aligned with what the project actually needs — and keeps CLAUDE.md and the context window small, because skill bodies load only when invoked (see §15).

Available packs

Installation and onboarding behavior

  • Default: no optional pack is installed. Onboarding asks which packs to install; if the repository clearly indicates applicability (e.g. a frontend for ui-qa), Claude suggests the pack but installs only on confirmation.
  • Selected packs are copied from skills/optional/<name>/ to .claude/skills/<name>/ and adapted to the project (commands, paths, doc locations). Exception: run-and-verify is generated from templates/run-and-verify.md.template instead of copied — see §18.
  • Installed and declined packs are recorded in decisions/answers.md, so declined packs are not re-proposed.
  • The generated CLAUDE.md lists installed packs by name with a one-line purpose; skill bodies are never pasted into it.
  • Packs can be added later at any time with the same copy-and-adapt rule.

Usage example

In a project that installed decision-log, after a design review settles on PostgreSQL over DynamoDB: "Use the decision-log skill to record that we chose PostgreSQL, why, and what we rejected." Claude drafts the entry, proposes it, and appends it to decisions/architecture-decisions.md only after approval.

Structure of every pack

Each SKILL.md documents: purpose; when to use; when not to use; required inputs; output artifact; safety constraints; and a short procedure. All packs are advisory or permission-gated — none mutates git state, external systems, or memory without explicit developer approval.

18. Run and verify skill

Implementation quality depends on knowing how to run the project and validate behavior. Instead of rediscovering that in every session, onboarding captures it once: when the run-and-verify pack is selected (§17), Claude generates a project-specific .claude/skills/run-and-verify/SKILL.md from templates/run-and-verify.md.template. The generated skill records the project's real commands and verification steps, and the implementer and reviewer run applicable checks through it.

How generation works

  1. Inspect first. Claude fills the template placeholders (DEV_SERVER_COMMAND, TEST_COMMAND, TARGETED_TEST_COMMAND, LINT_COMMAND, TYPECHECK_COMMAND, BUILD_COMMAND, REQUIRED_SERVICES, ENV_VAR_NAMES, UI_VERIFICATION, API_VERIFICATION) from the Phase 1 repository inspection — package.json scripts, Makefile, CI config, existing docs — and confirms the inferred commands with the developer.
  2. Ask only the gaps. The remaining questions come from questions.md §15: dev server and URL, targeted test command, build command, required services, environment variable names, and how to verify UI/API behavior. Test/lint/typecheck answers from §7 are reused, not re-asked.
  3. Never invent. Anything that cannot be confirmed is recorded as TODO: ask the developer. A skill with honest TODOs is useful; a skill with invented commands is dangerous.

What the generated skill contains

  • A commands table: dev server, all tests, targeted tests, lint, typecheck, build.
  • Required services (databases, queues, containers) and how to start them locally.
  • Required environment variables — names only, with a pointer to where they are documented (e.g. an .env.example file).
  • UI and API/CLI verification recipes: routes to open, expected states, example requests and responses, smoke commands.
  • A verification procedure: narrowest check first, then broader checks, then runtime behavior against the spec's acceptance criteria, then an honest report (what ran, what passed, what failed, what could not be verified and why).

Examples by project type

Project typeTypical commandsTypical verification
Frontend (e.g. Vite/Next.js)npm run dev · npm test · npx vitest run path/to.test.ts · npm run lint · npm run buildOpen the affected route on the dev server, compare against the spec's expected visual state; run the ui-qa skill if installed.
Backend API (e.g. Express/FastAPI)docker compose up -d db · npm run dev / uvicorn app:app --reload · targeted test per routeSend an example request (curl/HTTP client) to the changed endpoint and compare the response with the acceptance criteria.
Librarynpm test / pytest · npm run typecheck · npm run buildNo server to run: targeted tests plus a small consumer snippet exercising the public API that changed.
CLI toolgo build ./... / cargo build · unit testsInvoke the built binary with the spec's example arguments and compare stdout/exit codes.
MonorepoPer-package commands (pnpm --filter <pkg> test, turbo run lint)Run the affected package's checks first, then the workspace-level checks; the skill names which packages map to which commands.

These are illustrations — the generated skill always uses the project's verified commands, never assumed ones.

Review integration

When the skill is installed, the review flow depends on it: the implementer runs validation through it before moving a task to review (agents/implementer.md), and the reviewer verifies that applicable checks were run through it, with the evidence — commands run, results, unverified items — recorded in review.html (agents/reviewer.md, review checklist §4). TODO entries in the skill surface as explicitly unverified items, never as silent gaps.

Secrets

The generated skill never stores secret values. Environment variables appear by name only; values come from the developer's local environment or the project's documented mechanism. This applies to verification evidence too: never paste tokens, credentials, or secret-bearing output into specs, reviews, or the skill itself. Verification runs against local/test environments — never production without explicit instruction.
