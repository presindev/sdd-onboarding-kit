The workflow

Spec first. Code second. Always.

When the harness is installed, Claude Code follows a structured loop — creating and getting approval for a specification before a single line of production code is written.

  1. Select a task. The developer picks or describes what to build.
  2. Spec is created. Claude writes requirements, design, and implementation tasks — no code yet.
  3. You review and approve the spec before anything is implemented.
  4. Claude implements strictly against the approved spec — no scope creep.
  5. Tests and validation run automatically.
  6. Traceability review verifies requirements, design, tasks, code and tests are aligned.
  7. Documentation is updated when the change affects docs — task stays open until docs are current.
  8. Task is marked done or returned for correction.
Installation

Up and running in one prompt

Copy this kit into your project folder, then open Claude Code and run:

Read `sdd-onboarding-kit/instructions.md` and configure this repository to use Spec Driven Development. Ask me all necessary questions before making project-specific decisions.

Claude Code will inspect the repository, ask you the configuration questions from questions.md, and generate a complete project-specific SDD harness — adapted to your commands, stack, and preferences.

Recommended defaults available. Accept the default profile to skip twenty configuration questions and get started immediately. You can always customise later.

Safety model

Human-gated by design

  • Spec approval is required before any implementation begins — no exceptions.
  • Hooks are optional and disabled by default — examples only until you explicitly enable them.
  • Autonomy is opt-in: monitoring CI is fine; deploying, merging, or marking tasks done is never autonomous.
  • Git mutations are permission-gated: no commit, push, or force-push without explicit instruction.
  • No secrets are stored in any memory layer — the failure-learning pack proposes entries, never writes them.
  • MCPs are off by default — none configured without your explicit approval.
  • Playwright is scoped to its agent — browser tools never load into the main conversation.
Open source

Contributions welcome

Fork, improve, and send a PR. Keep changes focused — one concern per pull request.

  • Keep all files in English.
  • Template files under templates/ use {{PLACEHOLDER}} syntax.
  • Hook scripts must be POSIX-compatible bash and degrade gracefully without jq.
  • Agent files must follow the frontmatter format (name, description, tools).
